# 中文论文逻辑检查技能

对中文论文进行最终的一致性与逻辑核对。这是"红线审查"，只抓致命问题。

## 核心原则

**高容忍度，仅报错。**

- 默认假设文档已经经过多轮修改，质量较高
- 只在遇到阻碍读者理解的逻辑断层、引起歧义的术语混乱、或严重语法错误时才提出意见
- 对于"可改可不改"的风格问题直接忽略
- 不要通过挑刺来体现存在感

## 适用场景

- 论文终稿提交前的最终检查
- 用户要求"检查逻辑""看看有没有矛盾""核对一致性"
- 答辩/投稿前的最后一道质量关

## 检查维度

### 1. 致命逻辑
- 是否存在前后完全矛盾的陈述？
- 论证链条是否完整？有没有逻辑跳跃导致结论不成立？
- 引用是否支撑了论点？有没有引用的数据和论点对不上？

### 2. 术语一致性
- 核心概念是否在没有说明的情况下换了名字？
- 同一个方法/模型/数据集是否全文使用统一的名称？
- 缩写首次出现时是否给出了全称？

### 3. 数据一致性
- 正文中的数据是否与表格/图表中的数据一致？
- 百分比和绝对数值是否对应？
- 同一指标在不同位置引用时数值是否相同？

### 4. 严重语病
- 是否存在导致句意不清的表达？
- 主语是否缺失导致歧义？
- 指代是否明确（"这种方法"指的是哪种）？

## 工作流程

### 第一步：通读全文

用 `read` 读取完整文档。理解全文论证结构和核心主张。

### 第二步：逐章检查

对每个章节，按4个维度检查。只记录必须修改的问题。

### 第三步：交叉核对

- 引言中声称的贡献 vs 实验部分的实际验证
- 摘要中的关键数据 vs 正文中的具体数字
- 方法描述 vs 实验设置的一致性
- 图表标题和引用是否与正文描述一致

### 第四步：输出

如果没有"必须修改"的错误：

```
[检测通过，无实质性问题]
```

如果有问题，用中文分点简要指出：

```
## 逻辑检查结果

### 问题 1：[维度名] — [严重程度]
- **位置**：第 X 节第 Y 段
- **问题**：一句话说明
- **原文**：「相关引用」
- **建议**：一句话修复方向（不说具体怎么改，只说改什么）

### 问题 2：...
```

不要长篇大论。每个问题控制在 2-3 行内。

## 严重程度定义

| 严重程度 | 含义 | 处理要求 |
|----------|------|----------|
| **FATAL** | 论证链断裂，数据矛盾，会导致审稿人直接拒稿 | 必须修复 |
| **WARNING** | 术语不一致，指代模糊，可能引起审稿人质疑 | 建议修复 |
| **INFO** | 轻微不一致，不影响理解 | 可以忽略 |

只报告 FATAL 和 WARNING 级别的问题。INFO 级别不报告。

## 写作规范参考

逻辑检查不涉及风格和表达问题，但如果发现明显的翻译腔或 AI 味导致句意不清，可以在严重语病维度中报告。完整写作规范见 `/skill:plain-chinese-writing`。

## 反面模式

- 不要报告风格问题（"这段可以更简洁"）
- 不要建议换词（"可以用XX替换YY"）
- 不要提出"听起来更好"的建议
- 不要过度解读作者的意图
- 不要把标点风格问题当成逻辑问题报告
- 如果不确定是否有问题，不报告
