# 中文 Markdown 写作技能

直接起草或修改用户已授权范围内的中文 Markdown 文档。默认由写作 agent 完成一次聚焦处理，不要求逐段确认。只有用户明确要求逐段审查时，才进入交互式逐段模式。写作时遵循中文平直写作规范。

## 适用场景

- 用户要求起草或修改中文论文、报告、技术说明等 Markdown 文档
- 用户明确要求一段一段写或逐段确认

## 默认直接工作流

对于“润色摘要”“修改这一节”等普通请求：

1. 读取一次指定文本及修改所必需的相邻上下文。成功读取已经完整到达目标末尾时，不在修改前换 selector 重读；只有明确出现截断标记或请求范围不完整时才重读。
2. 在用户已经授权的范围内直接应用修改。
3. 对语义、结构、引用和格式做一次聚焦复核。
4. 汇报实际限制。不能仅因加载了本技能而要求用户重新授权或确认。

修改前记录限定、模态、范围、否定、方向、数字和单位、引用和标识符，以及 LaTeX 数学、交叉引用、命令和结构等语义锚点。保留这些锚点不等于冻结其余措辞。用户明确要求修改且原文存在可修问题时，应至少应用一处不改变含义的具体修改。某个候选会改变锚点时，只放弃该候选，继续寻找锚点之外的安全词语或结构修改。确实没有安全改进时才保持原文件不变，并直接说明限制，不增加验证读取。

逐个独立评估修改候选。准备保持原文不变前，至少检查一次不影响锚点的重复词、语法、搭配或冗余表达。一个候选不安全，只淘汰该候选。例如，相邻词语意外重复时，只要保留一次不改变含义，就可以删除重复项。只有 edit 成功后才执行一次验证读取。

只有用户明确要求逐段批准，或者存在会实质改变结果的歧义时，才使用下方的精细模式。

## 文档结构

- `##` 到 `#####`：章节标题，不要修改
- `######`：段落节点。标题=主题句（≤50字符），正文长度服从语义完整和章节需要
- HTML 注释 `<!-- description: ... -->`：描述每节应覆盖的内容

## 输入文件

| 文件 | 用途 |
|------|------|
| paper.md | 主文档，扫描空章节，读取上下文 |
| storyline.md | 研究叙事、核心想法、方法 |
| .pi/research/literature.md | 文献摘要 |
| .pi/research/papers/ | 下载的论文，只读当前段落相关的 |

## 可选精细模式

### 第一步：生成整体大纲

1. 读取 storyline.md 和已有 paper.md 的结构（第 2-5 级标题及其 description 元数据）。
2. 为每个空的第 5 级章节提出主题句（每个≤50字符），列在父标题下面。
3. 把大纲展示给用户。问："这是段落大纲，是否接受？需要修改还是重新生成？"
4. 仅在用户明确要求精细模式时等待回复。

### 第二步：定位并提议段落

1. 从上到下读 paper.md。找到第一个没有第 6 级子节点的第 5 级章节。
2. 报告章节名称和 description 元数据。问："下一个小节是[标题]。是否在这里写一段？"
3. 仅在用户明确要求精细模式时等待确认。

### 第三步：收集上下文并起草

读取这些文件获取上下文：
- storyline.md
- .pi/research/literature.md
- 相关的 .pi/research/papers/ 文件
- 当前 paper.md（避免重复）

然后写一段（一个 `######` 节点）：
- 标题：大纲中批准的主题句（≤50字符）
- 正文：支撑文字，长度服从语义完整和章节需要，需要时加引用
- 不要修改第 2-5 级标题
- 用平直、准确的中文。简洁。不凑字数。不要机械拆句。

### 第四步：用户审查

用户明确要求精细模式时，问："接受、修改（给反馈）还是重写？"，并等待回复。普通直接工作流中，不额外暂停，直接完成用户要求的范围并汇总改动。

### 第五步：应用并循环

- 接受：用 edit 把新的 `######` 节点插入目标章节下面。回到第二步。
- 修改：按用户反馈修改，重新展示（回到第四步）。
- 重写：丢弃重写（回到第三步）。

精细模式下，第一、二、四步不能自动进入下一段。默认直接工作流在一次聚焦处理中完成用户要求的范围。

## 中文写作规范

写作时遵守 `plain-chinese-writing` 技能中的相关规范。以下是要点提醒。

### 标点规则
- 破折号完全禁用
- 冒号限制使用（列举三项以上、引出定义时可用）
- 括号限制使用（缩写定义、编号可保留）
- 黑体限制使用（关键词定义、列表标签可保留）

### 语言风格
- 平直、准确、简洁
- 多用动词，少用抽象名词
- 语义完整，长短相间
- 注意中文特征：主题突出、意合衔接、主动优先
- 不为了规避标点规则而机械拆句
- 保留必要的限定、转折和承接信息
- 不机械使用"不在于……而在于……"转折模板
- 直接写具体困难、原因和后果，少用模板化转折
- "的"字不要连用三个以上

### 翻译腔和 AI 味
- 不堆叠名词，不滥用"进行""实现""基于"
- 不用英文语序，不用过多框式介词
- 不写空泛开头、泛化评价、套话结尾

## 反面模式

- 不要把精细模式强加给普通直接修改请求
- 精细模式中不要一次写多段
- 精细模式中不要跳过大纲生成
- 精细模式中不要在用户确认前插入内容
- 不要修改章节标题
- 不要把所有段落写成相同的短句结构
- 不要把每个关键判断都写成"不是……而是……"或"不在于……而在于……"
