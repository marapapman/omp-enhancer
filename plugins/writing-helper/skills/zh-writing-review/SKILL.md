# 中文写作审查修复引导

审查完成后，引导用户逐项修复发现的问题。按严重程度排序，每个问题确认后再处理下一个。

## 前提条件

`.pi/research/checker_report.md` 必须存在。如果不存在，先运行 `/skill:zh-writing-checkers`。

## 调用方式

`/skill:zh-writing-review` — 在 `zh-writing-checkers` 完成后触发。写作或审查 agent 调用。

## 工作流程

### 第一步：读取审查报告

读取 `.pi/research/checker_report.md`。文件缺失或为空时，告诉用户先运行 `/skill:zh-writing-checkers`，然后停止。

### 第二步：按严重程度排序

收集报告中的所有问题。按以下顺序排列：

1. **CRITICAL** — 事实错误、矛盾、逻辑断裂、缺少必要章节
2. **IMPORTANT** — 论点不清、证据薄弱、结构性问题、翻译腔、AI 味
3. **MINOR** — 用词、格式、风格细节

同一严重程度内，保留报告中的原始顺序。

### 第三步：逐项处理问题（最多 20 项）

对每个问题，向用户展示：

```
---
### 问题 [N]：[维度名] — [严重程度]
- **问题描述**：[报告中的具体文字或问题]
- **修改建议**：[一条具体的修改方案]
---
```

然后请用户选择：

- **接受** — 应用建议的修改
- **修改** — 用户提供替代方案，用替代方案
- **跳过** — 跳过此问题（记录跳过原因）

规则：
- 展示下一个问题前等用户输入，不要自动前进。
- 不要自动应用修改，必须先问。
- 如果建议的修改会改变文档的核心论点或论题，在建议行后面加上 "⚠️ 需要作者决定"。
- 一个会话最多处理 20 个问题。到 20 个后告诉用户："已达到 20 项上限。重新运行审查器可以处理剩余问题。"

### 第四步：记录每个处理结果

每个问题处理完后，追加到 `.pi/research/review_log.md`：

```
## 问题 [N]：[维度名] [严重程度] [描述]
- 处理结果：已修复 | 已修改 | 已跳过
- 变更内容：[改了什么，或跳过原因]
```

日志文件不存在则创建。追加，不覆盖。

### 第五步：所有问题处理完毕后

建议重新运行审查器验证修复效果：

> 所有问题已处理。运行 `/skill:zh-writing-checkers` 验证修改效果。

如果会话达到了 20 项上限，补充：

> 重新运行审查器后，可以再运行 `/skill:zh-writing-review` 处理剩余问题。

## 交互示例

```
---
### 问题 1：表达清晰度 — CRITICAL
- **问题描述**：第 3 节第 2 段使用了翻译腔表达"实现了对数据的有效处理"。
- **修改建议**：改为"有效处理了数据"。
---
接受 / 修改 / 跳过？
```

用户回复："修改 → 改为'处理了数据，效果不错'"

```
## 问题 1：表达清晰度 CRITICAL 翻译腔表达"实现了对数据的有效处理"
- 处理结果：已修改
- 变更内容：改为"处理了数据，效果不错"。
```

## 写作规范参考

审查修复时遵守 `/skill:plain-chinese-writing` 的完整规范。建议的修改方案应符合中文平直写作要求，避免引入新的翻译腔或 AI 味。
