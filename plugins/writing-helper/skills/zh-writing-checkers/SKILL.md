# 中文写作审查器

## 目的

按 7 个质量维度审查中文文档（论文、提案、报告），输出结构化、可操作的反馈。每个发现引用具体证据（章节、段落、行号）。不捏造问题，只说文档实际存在的问题。

## 工作流程

### 第一步：通读全文

用 `read` 加载完整文档后再开始审查。如果是多文件目录，用 `grep`/`find` 定位主内容文件。理解完整论证后再评估任何维度。

### 第二步：按顺序逐维度审查

对下面的每个维度，输出一个审查块。不要跳过维度，不要调换顺序。

| # | 维度 | 核心问题 |
|---|------|----------|
| 1 | **问题** | 问题定义清楚吗？重要性有证据支撑吗？关联到具体需求吗？ |
| 2 | **新颖性** | 贡献真正新颖吗？与已有工作的区分明确吗？指出了已有工作的具体不足吗？ |
| 3 | **深度** | 有真正的技术深度吗？解决了实际的技术难点吗？不只是调参？ |
| 4 | **逻辑** | 论证内部一致吗？每个论点有支撑证据吗？有矛盾或推理跳跃吗？ |
| 5 | **清晰度** | 术语都定义了吗？引用准确吗？语言精确吗？有翻译腔或 AI 味吗？读者能跟上吗？ |
| 6 | **评估** | 实验映射到研究问题了吗？基线合理吗？指标适合论点吗？ |
| 7 | **数据** | 数据/代码引用真实可复现吗？有复现路径吗？报告的数字可验证吗？ |

### 第三步：每个维度的输出格式

```
### [维度名]
- **状态**：通过 | 发现问题
- **问题**：
  - [CRITICAL] 描述 — 第 X 节/第 Y 段。「具体引用」
  - [IMPORTANT] 描述 — 第 X 节/第 Y 段。「具体引用」
  - [MINOR] 描述 — 第 X 节/第 Y 段。「具体引用」
```

严重程度定义：
- **CRITICAL** — 根本性缺陷，影响整个论点。必须修复。
- **IMPORTANT** — 影响严谨性或完整性。应该修复。
- **MINOR** — 改进建议。有则更好。

### 第四步：最终摘要

7 个维度全部审查后，输出：

```
## 摘要

| 维度 | 状态 | 问题数 |
|------|------|--------|
| 问题 | 通过 / 发现问题 | N 个 |
| 新颖性 | 通过 / 发现问题 | N 个 |
| 深度 | 通过 / 发现问题 | N 个 |
| 逻辑 | 通过 / 发现问题 | N 个 |
| 清晰度 | 通过 / 发现问题 | N 个 |
| 评估 | 通过 / 发现问题 | N 个 |
| 数据 | 通过 / 发现问题 | N 个 |

**总评**：就绪 | 需要修改 | 严重问题
```

总评规则：
- **就绪** — 所有维度通过，或只有 MINOR 问题。
- **需要修改** — 存在 IMPORTANT 问题，但没有 CRITICAL。
- **严重问题** — 存在 CRITICAL 问题。应醒目报告；本检查不阻止后续编辑、工具调用或会话结束。

## 中文写作专项检查

在清晰度维度中，额外检查以下项目。完整规范见 `plain-chinese-writing` 技能。

### 翻译腔检查
- 名词堆叠（"实现了对……的有效识别"）
- 过度使用"进行""实现""基于"
- 英文语序（"这是一个被……的方法"）
- 英式长定语（"一个…的…的…"连续修饰结构）
- 抽象主语过多
- 被动句过多（中文写作少用"被"字句）
- 框式介词过多（"在……方面""对于……来说"）
- "的"字连用过多（连续三个以上）
- 机械列举格式（"首先…其次…最后…"）。只有列举确实造成模板感或割裂感时才报告

### AI 味检查
- 空泛开头（"近年来，随着……的快速发展"）
- 过度结构提示（"本文将从以下几个方面展开"）
- 机械转折（"因此，问题不在于……而在于……"，"核心困难不在于……而在于……"）
- 泛化评价（"具有重要意义"）
- 套话结尾（"未来仍需进一步探索"）
- 情感渲染性表达（"毋庸置疑""令人惊叹""颠覆性"）
- 绝对禁用词（赋能、抓手、闭环、沉淀等）
- 需要检查的词（鲁棒性、显著性、前沿、创新性，检查是否具体、必要）

### 语体规范检查
- 是否存在陈旧公文腔（无故用"拟"代"旨在"，用"系"代"是"）
- 是否存在口语残留（"我们觉得""效果变好了"）
- 连接词是否机械堆砌，语义衔接是否自然
- 是否机械使用"因此……不在于……而在于……"这类转折模板。只有确实在纠正误解时才允许保留
- 段落是否有清楚主线。不要要求每段只能有一个信息点，必要的限定、证据、转折和小结可以留在同一段
- 段落内句间衔接是否自然。不要把自然中长句误判为问题
- 中英文混排空格是否规范（中英文之间加空格）

### 格式问题
- 完全禁用的破折号（——）
- 非必要场景的冒号（列举三项以上和引出定义可保留）
- 括号内塞长句的滥用（缩写定义和编号可保留）
- 代替表达强度的黑体（关键词定义和列表标签可保留）
- 不必要的成语
- 不合适的项目符号或编号列表。只有列表破坏论证连贯性时，才建议改为段落

### 中文风格问题的严重程度

- 破折号是硬性格式问题，必须报告
- 冒号、括号、黑体的滥用只有确实滥用时才报告，合理使用不算问题
- 空泛开头、套话结尾、泛化评价、明显翻译腔、机械转折和机械列举只有在原文中有明确证据时才报告
- 普通风格偏好默认不超过 MINOR，不得因为句子不够短、段落不够规整或存在自然连接词而判为问题
- 不要把自然的中长句判为 AI 味。只有句子层级混乱、信息堆叠或影响理解时才报告
- 审查建议不得要求作者删除必要的限定、转折、因果和承接信息

## 审稿人视角审视（附加维度）

在7维度审查之后，附加一轮审稿人视角的审视。这不是必须的维度，但如果文档是论文，应当执行。

### 审视原则
- 客观评估论文实际水平，精准定位不足，如实肯定贡献
- 区分"真正致命的问题"与"可以修复的小问题"，两者权重完全不同
- 评分忠实反映论文实际水平
- 省略无关痛痒的客套表述，直接切入核心判断

### 审视内容
- 社区贡献：论文是否为领域带来了实质性推进？
- 严谨性：核心主张是否有充分的实验支撑？基线是否齐全、版本是否对齐？消融实验是否覆盖了关键设计决策？
- 一致性：引言中声称的贡献在实验部分是否真正得到了验证？有没有被回避的核心问题？

### 可救性判断
对每个发现的问题，给出可救性判断：
- 可以在修订期内解决（表述问题、补充实验）
- 方法层面的结构性缺陷（难以靠补充实验弥补）

### 输出格式
```
### 审稿人视角
- **总评**：[一句话总结]
- **优势**：[1-3 点真正有价值的贡献]
- **致命问题**：[具体到实验设置、论证环节或表述缺陷，不接受泛泛而谈。若无则如实说明]
- **可修复问题**：[列出可以在修订期内解决的问题]
- **评分**：[1-10 分] + 一句话评分依据
```

## 规则

1. 先通读全文，再开始审查。
2. 按顺序逐维度审查，一次一个。
3. 每个发现引用具体证据。用章节号、段落号或行范围。尽量引用原文。
4. 不捏造问题。一个维度没问题就报告通过。
5. 标记缺失前，先彻底搜索确认确实缺失。
6. 一个问题一条，不把多个问题打包。
7. 严重程度要准确。CRITICAL 只用于根本性缺陷。
8. 保留语义锚点。不得仅为改善文风而建议删除频率和强度限定、模态、范围、否定、比较和因果方向、数字和单位、引用和标识符，或 LaTeX 数学、交叉引用、命令和结构。同时拿到原文和修改稿时，只比对一次并报告漂移。

## 输出位置

只有用户允许写报告时才写入 `.pi/research/checker_report.md`。只读审查直接在最终响应中返回同样的结构化报告。不创建 `.pi` 目录，也不为技能模板请求写权限。

## Pi 集成

- **使用**：通过运行时正常的技能加载方式读取 `zh-writing-checkers`，不要尝试不存在的斜杠命令。
- **使用工具**：`read`、`grep`、`find`。
- **输出**：允许写报告时使用报告路径；只读模式使用最终响应。不创建或修改其他文件。
